Architecture

현재 상위 구조

Assistant App이 항상 중심이며, 실행만 두 백엔드 중 하나로 위임됩니다. 

사용자 / 관리자 / Webex
        │
        ▼
  assistant_app/
    ├─ orchestrator
    ├─ policy_evaluator
    ├─ webex_gateway
    └─ approval_manager
        ├─ separated → device_executor/
        └─ all_llm   → direct_tool_adapter/
                        ▼
             Webex cloud xAPI
  • admin_page/ 는 얇은 정적 브라우저 표면입니다.
  • 실제 장치 경로는 로컬 RoomOS가 아니라 Webex cloud xAPI 입니다.
  • 정책과 승인 판단은 assistant_app 에 남아 있습니다.

Flows

메시지와 승인 흐름

로컬 디버그, 실제 Webex 메시지, 카드 클릭 모두 현재 구현에 맞게 분리되어 있습니다.

  • /debug/messages 는 가장 빠른 로컬 진입점입니다.
  • /webhooks/webex/messages 는 raw 서명 검증 후 실제 메시지를 fetch 합니다.
  • 승인/선택 카드는 /webhooks/webex/attachment-actions 와 fetch-by-id semantics 를 사용합니다.
  • /admin-page 자체는 열려 있지만 보호 데이터는 관리자 세션 후에만 로드됩니다.

Use

지금 바로 쓰는 방법

기본 개발 경험은 mock-first 이므로 외부 자격 증명 없이도 먼저 시작할 수 있습니다. 

python3 -m venv .venv
.venv/bin/python -m pip install -e ".[dev]"
.venv/bin/python -m uvicorn assistant_app.main:app --reload
  • /admin-page/webex-test 에서 Webex 런타임과 디버그 호출을 확인할 수 있습니다.
  • /debug/webex/simulate-message 는 WEBEX_MOCK_MODE=true 일 때만 동작합니다.
  • 읽기 기능은 바로 실행되고 대부분의 변경 기능은 기본적으로 승인 필요입니다.
현재 범위와 한계

현재 런타임 분석 제공자로 실사용 가능한 것은 rule_based 와 ollama 입니다. booking CRUD, 로컬 RoomOS 전송, 재시작 안전한 세션 메모리는 아직 범위 밖입니다.
