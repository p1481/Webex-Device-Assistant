Flow

High-level architecture

Natural language is converted into canonical actions, policy/approval is applied, then Webex cloud xAPI executes the device request.

User/Webex/Debug API
  → FastAPI assistant_app.main
  → ProviderRegistry: rule-based or Ollama
  → Orchestrator: intent + slots + cards
  → PolicyStore / ApprovalStore
  → ModeRouter
      separated: DeviceExecutor → DeviceClient → Webex cloud xAPI
      all-LLM: AllLlmToolRuntime → execute_device_action → DirectToolAdapter → DeviceClient

Service

FastAPI runtime

assistant_app.main:app registers health, debug, Webex webhook, admin JSON, and admin-page routes.

  • Health: GET /healthz
  • Local test: POST /debug/messages
  • Admin UI: GET /admin-page
  • Webex ingress: /webhooks/webex/messages, /webhooks/webex/attachment-actions

LLM

Provider integration

Runtime analysis currently supports deterministic rule parsing and local Ollama parsing.

  • DEFAULT_PROVIDER=rule-based: predictable parser.
  • DEFAULT_PROVIDER=ollama: local LLM structured proposal flow.
  • DEFAULT_PROVIDER_MODEL and DEFAULT_PROVIDER_BASE_URL configure model/runtime.
  • all-LLM mode still validates the canonical request before executing the tool.

Execution

Separated vs all-LLM

Both modes preserve policy and approval. Separated mode executes directly through the deterministic executor; all-LLM mode requires one validated tool call.

  • DEFAULT_EXECUTION_MODE=separated is the safest default.
  • DEFAULT_EXECUTION_MODE=all-llm uses execute_device_action.
  • Reboot and factory reset are separated-mode only.

Configuration

Environment variables

Mock mode defaults allow local startup before real Webex credentials are available.

  • WEBEX_MOCK_MODE, DEVICE_MOCK_MODE: default true.
  • WEBEX_BOT_TOKEN, WEBEX_BOT_PERSON_ID, WEBEX_WEBHOOK_SECRET: required for real Webex ingress.
  • WEBEX_TOKEN_MANAGER_BASE_URL, WEBEX_TOKEN_MANAGER_API_KEY: required for real device execution.
  • DEFAULT_TARGET_DEVICE: fallback device name.
